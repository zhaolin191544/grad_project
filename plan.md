## 项目重写 Plan：IFC 三维模型协作平台

### 定位

从"IFC 文件查看器"升级为**"IFC 模型管理与智能分析平台"**——既满足毕设对三维渲染+AI+全栈的要求，又能在简历上体现 Next.js 全栈能力、AI 集成经验、3D 可视化技术深度。

------

### Phase 1：基础架构（约 3-4 天）

| 模块                   | 内容                                                         |
| :--------------------- | :----------------------------------------------------------- |
| **Next.js 项目初始化** | App Router, TypeScript, Tailwind CSS, ESLint                 |
| **数据库**             | PostgreSQL + Prisma ORM，设计核心表：`User`, `Project`, `Model`, `ModelComment` |
| **认证**               | NextAuth.js（GitHub/邮箱登录），Session 管理                 |
| **文件存储**           | IFC 文件上传到本地 `/uploads` 目录（或对接 S3/MinIO），数据库存元数据 |
| **基础布局**           | 响应式 Dashboard 布局，侧边栏导航，亮色主题                  |

**简历亮点**：全栈架构设计、Prisma ORM、NextAuth 认证

------

### Phase 2：IFC 三维查看器核心（约 4-5 天）

| 模块                    | 内容                                                         |
| :---------------------- | :----------------------------------------------------------- |
| **Three.js 渲染器封装** | 将现有渲染逻辑拆分为 React 组件/hooks：`useThreeScene`, `useIFCLoader`, `useOrbitControls` |
| **IFC 加载与解析**      | 复用 web-ifc-three，支持从服务器加载已上传的 IFC 文件        |
| **交互功能迁移**        | 模型选择高亮、元素属性面板、线框/X-Ray/剖切面                |
| **视图控制**            | 预设视角（顶视/前视/ISO）、相机位置显示、截图导出            |
| **构件树**              | 解析 IFC 空间结构，树形展示楼层→空间→构件层级，点击定位      |

**简历亮点**：Three.js/WebGL 3D 渲染、IFC/BIM 领域知识、React Hooks 封装复杂状态

------

### Phase 3：AI 智能分析模块（约 3-4 天）

这是毕设和简历的**最大加分项**。

| 模块            | 内容                                                         |
| :-------------- | :----------------------------------------------------------- |
| **AI 对话面板** | 右侧抽屉式聊天界面，支持流式输出（SSE）                      |
| **模型问答**    | 用户可以针对当前 IFC 模型提问，如"这个建筑有几层？""列出所有门窗构件""一层的面积是多少？" |
| **实现方式**    | Next.js API Route 调用大模型 API（OpenAI/Claude），将 IFC 解析出的结构化数据（空间结构、构件列表、属性）作为 context 注入 prompt |
| **智能指令**    | 支持自然语言操控视图，如"高亮所有墙体""切换到顶视图""打开X光模式" —— AI 返回结构化 JSON 指令，前端执行 |

**简历亮点**：AI/LLM 集成、Prompt Engineering、SSE 流式传输、语义-操作映射

------

### Phase 4：项目管理与协作（约 2-3 天）

| 模块         | 内容                                                  |
| :----------- | :---------------------------------------------------- |
| **项目空间** | 用户创建项目，上传多个 IFC 模型到项目中               |
| **模型列表** | 卡片式展示，显示缩略图、文件大小、上传时间、构件数量  |
| **构件批注** | 选中 3D 构件后可添加文字批注，批注数据存入 PostgreSQL |
| **历史记录** | 记录用户的查看、批注、AI 对话历史                     |

**简历亮点**：CRUD 全流程、关系型数据设计、用户体验设计

------

### Phase 5：性能优化与论文素材（约 2 天）

| 模块           | 内容                                                 |
| :------------- | :--------------------------------------------------- |

| **性能仪表盘** | 页面内显示渲染帧率、模型加载时间、AI 响应延迟        |

**论文亮点**：有量化的性能评估数据，满足"系统集成与性能评估"要求

## 进阶模块

### 1. IFC 模型合规性检查引擎（规则引擎）

这是 BIM 行业的真实痛点，学术含量高。

| 内容                | 说明                                                         |
| :------------------ | :----------------------------------------------------------- |
| **规则 DSL**        | 设计一套声明式规则语法，如 `IfcDoor.height >= 2100mm`、`IfcWall.fireRating == "2h"` |
| **内置规则集**      | 预设建筑规范检查：门高是否达标、走廊宽度、楼梯踏步尺寸、防火分区面积 |
| **自定义规则**      | 用户可在界面上用表单/代码编辑器编写自定义检查规则            |
| **检查报告**        | 运行检查后生成报告，不合规构件在 3D 场景中标红高亮，列表展示违规详情 |
| **AI 辅助生成规则** | 用户用自然语言描述规范要求，AI 自动生成规则 DSL              |

**为什么有深度**：涉及 DSL 设计、AST 解析执行、IFC 属性深度遍历，论文可以单独写一章。简历上体现"领域建模"和"规则引擎"能力。

------

### 2. 模型版本对比（Diff）

类似 Git diff，但针对 3D 建筑模型。

| 内容               | 说明                                                         |
| :----------------- | :----------------------------------------------------------- |
| **结构对比**       | 上传同一建筑的两个版本 IFC，自动对比：新增/删除/修改了哪些构件 |
| **属性对比**       | 同一构件的属性变化（材质变了、尺寸改了）                     |
| **3D 可视化 Diff** | 场景中用颜色区分：绿色=新增，红色=删除，黄色=修改            |
| **变更统计**       | Dashboard 展示变更数量、影响楼层、变更类型分布图表           |

**为什么有深度**：涉及 IFC GlobalId 匹配算法、属性树递归对比、Three.js 多模型叠加渲染。这个功能市面上成熟开源方案很少，做出来就是亮点。

------

### 3. 高级渲染管线

从"能看"到"好看"，体现 WebGL 功底。

| 内容                           | 说明                                                         |
| :----------------------------- | :----------------------------------------------------------- |
| **SSAO（屏幕空间环境光遮蔽）** | 增强模型层次感和深度感，用 Three.js EffectComposer 后处理实现 |
| **Edge Detection 描边**        | 建筑图纸风格的线稿渲染，用 Sobel/Laplacian 后处理 shader     |
| **剖面填充**                   | 剖切面处自动填充截面图案（实心填充或材质纹理），不只是"切开"而已 |
| **热力图着色**                 | 根据构件属性值（面积、造价、能耗）动态映射颜色渐变到模型表面 |
| **渲染模式切换**               | 一键切换：真实感 / 线稿 / 热力图 / X-Ray，做成一个渲染模式系统 |

**为什么有深度**：直接涉及 GLSL shader 编程和渲染管线定制，论文里可以放 shader 代码和渲染对比截图，技术含量一目了然。

------

### 4. 空间分析与工程量统计

把 IFC 从"可视化"推向"数据分析"。

| 内容                | 说明                                                         |
| :------------------ | :----------------------------------------------------------- |
| **工程量提取**      | 自动统计：墙体总面积、混凝土体积、门窗数量、管线长度等       |
| **按楼层/类型聚合** | 数据用 ECharts/Recharts 图表展示，支持按楼层、构件类型、材质分组 |
| **造价估算**        | 用户设定单价，系统自动计算各分项和总造价                     |
| **导出报表**        | 生成 Excel/PDF 工程量清单                                    |
| **AI 数据洞察**     | 将统计数据喂给 AI，让它生成分析摘要："该建筑主体结构混凝土用量约 XXm³，与同类型建筑相比偏高 15%，建议优化..." |

**为什么有深度**：这是 BIM 的核心价值——"信息"。从几何渲染深入到数据挖掘，论文可以体现"模型→数据→决策"的完整链路。

------

### 5. 实时多人协作（WebSocket）

类似 Figma 的多人实时体验。

| 内容         | 说明                                                   |
| :----------- | :----------------------------------------------------- |
| **实时光标** | 其他用户的鼠标位置实时显示在 3D 场景中（带用户名标签） |
| **选择同步** | A 用户选中某个构件，B 用户能看到高亮                   |
| **批注协作** | 多人同时批注，实时可见                                 |
| **在线状态** | 显示当前在线查看该模型的用户列表                       |
| **技术方案** | Next.js + Socket.io 或使用 Liveblocks/PartyKit         |

**为什么有深度**：WebSocket 状态同步、冲突处理、3D 场景中的多用户状态管理，简历上直接体现"实时协作系统"经验。

------

### 6. AI 深度集成（超越简单问答）

| 内容                    | 说明                                                         |
| :---------------------- | :----------------------------------------------------------- |
| **多轮对话 + 工具调用** | AI 不只是回答问题，而是能调用"工具"——查询构件、执行统计、修改视图、运行合规检查 |
| **Function Calling**    | 定义 tools schema，让 AI 自主决定调用哪个函数，形成 Agent 模式 |
| **RAG 模式**            | 将 IFC 属性数据向量化存入 pgvector，AI 基于检索增强回答      |
| **对话示例**            | "帮我检查所有防火门是否符合规范" → AI 自动调用合规检查引擎 → 返回结果并在 3D 中标注 |

**为什么有深度**：这不是简单的 API 调用，而是一个完整的 AI Agent 系统。论文可以讨论 Function Calling 架构设计和 prompt 策略。

## 调整后的优先级建议

```
必做（毕设核心）：
  Phase 1  基础架构
  Phase 2  IFC 查看器核心
  Phase 3  AI 智能分析（含 Agent/Function Calling）
  Phase 4  空间分析与工程量统计

强烈推荐（大幅加分）：
  Phase 5  合规性检查引擎
  Phase 6  高级渲染管线（至少 SSAO + 描边）

时间充裕再做：
  Phase 7  模型版本对比
  Phase 8  实时多人协作
```

"必做"部分已经覆盖了毕设的全部要求（全栈、3D、AI、性能评估）。"强烈推荐"部分是让项目从"毕设水平"跨入"简历亮点"的关键差异化。
